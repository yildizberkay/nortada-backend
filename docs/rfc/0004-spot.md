## 1. Summary

This RFC delivers the **spot** bounded context — the catalogue of watersports locations
that the whole product hangs off. A spot is a place (coordinates + name) enriched with the
"spot intelligence" that is Splash's actual differentiator: the shore orientation
(`shoreBearingDeg`), the sports it supports, the wind directions that make it good or
dangerous, its water type, and a skill hint. The RFC ships the `spot` table (DB name
`watersport_spot`) and the `user_favorite` table, the geo query stack (nearby / search /
detail), the favorites CRUD, an admin moderation surface, and the pipeline that *sources*
spots we don't have yet — a Trigger.dev task that pulls watersports POIs from OpenStreetMap
via the Overpass API and seeds them as `pending` for admin curation.

Two design choices dominate everything else. First, **no PostGIS and no third-party geo API**
([[decisions]] D-003): "nearby" is a cheap `(latitude, longitude)` bounding-box pre-filter
followed by an exact haversine distance computed *in SQL*, which both orders results and
enforces the true circular radius — sub-millisecond at our ~30k-row scale. Second, the only
clean, licence-compatible open dataset for watersports spots is **OpenStreetMap (ODbL)**;
commercial spot guides are comprehensive but ToS-protected and are used only as a curator's
reference, **never scraped or copied**. This RFC is also where two cross-cutting seams first
come alive: it introduces the **first Trigger.dev task** in the codebase (the canonical
task pattern for every later job), and it is the **first data-owning domain**, so it
implements the anonymous→Clerk **merge-reassign** hook (favorites are the first real user
data that must survive an account link — [[decisions]] D-008).

## 2. Motivation & Context

- **Problem.** The iOS app ([[../SPLASH-OVERVIEW]]) ships with a sample `Spot` list and no
  real backend. Before weather (RFC-0005), the decision engine, or activity tracking can
  reference a place, the backend needs a spot catalogue with the fields the app already
  expects, a way to *find* spots near a point, and — the hard part — a way to acquire spots,
  because **we start with an empty database**. The team does not have ~30k spots; it has to
  source them.
- **Background.** The geo *technique* was settled up front in [[decisions]] D-003 (plain
  Postgres, no PostGIS, no third-party geo API). The full data-model + sourcing design lives
  in [[spot-model-and-sourcing]]. The launch beachhead is the Turkish Aegean (Çeşme/Alaçatı,
  Gökova, Bodrum/Datça) with a globally-capable architecture ([[decisions]] D-007) — Alaçatı
  is a marquee PWA World Cup windsurf venue, so a Türkiye-first curation still touches the
  global goal. Auth (`c.var.user`, anonymous JWT vs Clerk, `isAdmin`) comes from RFC-0002.
  Review-driven refinements to this RFC are recorded in [[../otonom-kararlar]] §22–23.
- **Goals.**
  - A `spot` entity that consolidates PRD §13.2 and the app's `Spot` model, with
    `shoreBearingDeg` as the central piece of IP (it turns a live wind direction into an
    on/off/side-shore verdict at query time).
  - Nearby / search / detail endpoints backed by plain Postgres geo math, antimeridian-safe.
  - An OSM/Overpass ingest pipeline (Trigger.dev task) that seeds `pending` spots per country
    for admin curation, idempotent across re-ingests, ODbL-attributed.
  - "Suggest Spot" (user submission → `pending`) and an admin moderation surface
    (list-by-status, publish/reject/curate, request-ingest).
  - Favorites, plus the seams two later domains need: a minimal `SpotGeo` port for weather
    (RFC-0005) and a `MergeReassigner` for the auth merge (D-008).
- **Non-goals.**
  - **Weather / forecast** — RFC-0005 attaches to a spot's `uid`; spot only exposes the
    `SpotGeo` port and the favorites "hot set". No weather columns live on `spot`.
  - **Station mapping / station-confidence refs** — the PRD floated per-spot weather-station
    references; D-004 made Open-Meteo the single source with no station-confidence claim, so
    spot carries no station field.
  - **Automatic `shoreBearingDeg` derivation from coastline geometry** — deriving the shore
    normal from the nearest OSM `natural=coastline` tangent is a **P1** follow-up; for now
    curators set it by hand (see §14, §16 and [[../otonom-kararlar]] §23).

## 3. Scope (In / Out)

- **In:** the `spot` (`watersport_spot`) and `user_favorite` tables + the spot enums +
  `user.isAdmin`; the `packages/geo` helpers (`boundingBox`, `longitudeRanges`, `haversineKm`,
  `windSide`) and the `packages/overpass` client; `SpotRepository`, `FavoriteRepository`;
  `SpotService`, `FavoriteService`, `SpotIngestService` (+ the pure `normalizeOverpassElement`);
  the `spot-osm-ingest` Trigger.dev task; public routes (`/v1/spots`), admin routes
  (`/v1/admin/spots`), favorites routes (`/v1/me/favorites`); the `spot.module.ts` wiring; and
  the D-008 `favoriteReassigner` merge hook.
- **Out:** weather / forecast columns and endpoints (RFC-0005 — it mounts sub-paths under
  `/v1/spots/:uid`); the activity domain's own reassigner (RFC-0006 adds a second one to the
  same seam); coastline-tangent `shoreBearingDeg` auto-derivation (P1); dedupe/update diff
  cron against OSM (P1).

## 4. Domain Model & Ubiquitous Language

- **Spot.** A watersports location. Identity = internal `id` (never exposed) + public `uid`.
  Beyond place fields (`name`, `country`, `region`, `locality`, `latitude`, `longitude`) it
  carries curation intelligence: `waterType`, `supportedSports[]`, `skillSuitability`,
  `shoreBearingDeg`, `goodWindDirections[]`, `riskyWindDirections[]`, `hazards[]`.
- **`shoreBearingDeg`** (core IP). A 0–360° bearing = the **outward normal** of the shore,
  i.e. the direction pointing from land toward open water. It is *not* the coastline tangent
  (deriving it from a tangent must resolve the 180° land-vs-sea ambiguity). Combined with a
  wind's meteorological "from" direction it yields the on/off/side-shore verdict — offshore
  wind is the dangerous case (it blows a rider out to sea). Nullable until derived/curated.
- **`waterType`** — `sea | lake | bay | river | marina | open_coast`.
- **`source`** — provenance: `osm` (bulk-seeded from OpenStreetMap), `curated`
  (hand-authored/edited by an admin), `user_suggested` (from the app's Suggest Spot flow).
- **`status`** — the moderation state machine: **`pending`** (freshly ingested or suggested,
  invisible to the public) → **`published`** (visible in nearby/search/detail, favoritable) or
  **`rejected`** (hidden, kept for audit/dedupe). Public reads are always `published`-only;
  `pending`/`rejected` exist only behind admin.
- **WindSide** — the derived classification `onshore | cross-onshore | cross-shore |
  cross-offshore | offshore`, computed from `shoreBearingDeg` and a wind direction by
  `packages/geo` `windSide()`. A shared value used by both the spot and weather/decision logic.
- **Favorite.** A user⇄spot link (`user_favorite`), unique per `(userId, spotId)`. Two roles:
  the user's personal list, and — union across all users — the weather **"hot set"** (D-004),
  the distinct published spots worth pre-warming forecasts for.
- **MergeReassigner** — a domain's hook `(fromUserId, toUserId, tx) => Promise<void>` to move
  its user-owned rows during an anonymous→Clerk merge. Spot exposes `favoriteReassigner`; it is
  the first implementation of the D-008 seam.

## 5. Data Model (Drizzle)

All tables/enums/relations live in `src/db/schema.ts` (single source of truth); the migration
is generated with `npm run db:gen`. Coordinates are `doublePrecision` on purpose — `real`
(float4) loses meaningful longitude precision.

**Enums (`pgEnum`).**

| Enum | DB name | Values |
| --- | --- | --- |
| `waterTypeEnum` | `water_type` | `sea, lake, bay, river, marina, open_coast` |
| `spotSkillEnum` | `spot_skill` | `beginner, intermediate, advanced, all` |
| `compassDirectionEnum` | `compass_direction` | 16-point compass `N…NNW` |
| `spotSourceEnum` | `spot_source` | `osm, curated, user_suggested` |
| `spotStatusEnum` | `spot_status` | `published, pending, rejected` |
| `sportEnum` | `sport` | shared cross-domain enum (`windsurf, wingfoil, kitesurf, sailing, sup, kayak, other`, …) reused from RFC-0002 |

Good/risky wind directions are stored as the discrete 16-point compass; the *live*
on/off/side-shore verdict is **not** stored — it is derived from `shoreBearingDeg` at query
time so it always reflects the current wind.

**`spotTable` → `watersport_spot`** — the catalogue.

| Column | Type | Notes / rationale |
| --- | --- | --- |
| `id` | integer identity PK | internal key, never exposed |
| `uid` | text uuid (default) | public identifier used in URLs/payloads |
| `name` | text, not null | |
| `country` / `region` / `locality` | text, nullable | place hierarchy; `country` holds the ISO code for OSM-sourced rows |
| `latitude` / `longitude` | doublePrecision, not null | drives the bbox pre-filter + haversine |
| `waterType` | `water_type`, nullable | |
| `supportedSports` | `sport[]`, not null | array, filtered with `arrayContains` |
| `skillSuitability` | `spot_skill`, nullable | beginner/intermediate/advanced/all hint |
| `shoreBearingDeg` | `real`, nullable | **core IP**, 0–360° outward shore normal; null until curated/derived |
| `goodWindDirections` | `compass_direction[]`, nullable | curated |
| `riskyWindDirections` | `compass_direction[]`, nullable | curated |
| `hazards` | `text[]`, nullable | open set (`"offshore_wind"`, `"shallows"`, `"rocks"`…) → text, not an enum |
| `source` | `spot_source`, not null | provenance |
| `osmId` | text, nullable | `"{type}/{id}"` (e.g. `node/1`) when OSM-sourced; dedupe + future-update key |
| `suggestionNotes` | text, nullable | the suggester's free-text note for moderators ("launch is behind the pier…"); **never** in public responses — admin list only |
| `status` | `spot_status`, not null, default `pending` | moderation state |
| `createdBy` | integer → `user.id`, nullable | set for user-suggested spots |
| `createdAt` / `updatedAt` | timestamptz(3) | |

Indexes:
- `watersport_spot_lat_lon_idx` **`(latitude, longitude)`** — btree range-scans the leading
  column (latitude) for the bbox pre-filter; longitude is an in-index filter. Adequate at ~30k
  rows (D-003); revisit with `cube`/`earthdistance` + GiST only if the dataset grows large.
- `watersport_spot_status_idx` **`(status)`** — the public "published-only" and admin
  "by-status" filters.
- `watersport_spot_osm_id_key` — **partial** unique index `uniqueIndex(osmId) WHERE osm_id IS
  NOT NULL`. Enforces OSM dedupe while allowing many `NULL` `osmId`s (curated/user rows). This
  partiality has a direct consequence for the ingest upsert (§7): the `ON CONFLICT` arbiter
  must repeat the `WHERE osm_id IS NOT NULL` predicate or Postgres cannot select this index
  (error 42P10) — see §16.

**`favoriteTable` → `user_favorite`** — a user's favorited spots. It lives inside the *spot*
feature (not `platform/user`) so its FK to `spotTable` does not force a `platform → feature`
import — the forbidden direction. Columns: `id`, `uid`, `userId` → `user.id` (not null),
`spotId` → `spotTable.id` (not null), `createdAt`. Unique index `user_favorite_user_spot_key`
on `(userId, spotId)` makes "add favorite" idempotent and is the collision key the merge
reassign dedupes against (§7).

**`user.isAdmin`** (added to the existing `user` table) — `boolean("is_admin").notNull()
.default(false)`. Gates the admin moderation surface via the `requireAdmin` middleware; it is
the authorization anchor for this RFC and future admin features.

Units are canonical SI (D-006); bearings/coordinates are degrees, distances km at the API edge.

## 6. API Surface (routes + OpenAPI)

| Method | Path | Auth | Summary |
| --- | --- | --- | --- |
| GET | `/v1/spots/nearby` | user | Published spots within a radius, nearest first |
| GET | `/v1/spots/search` | user | Published spots by name (ILIKE) |
| GET | `/v1/spots/:uid` | user | Public spot detail (published only) |
| POST | `/v1/spots/suggest` | user (rate-limited) | Submit a spot → `pending` |
| GET | `/v1/admin/spots` | admin | List spots by moderation status |
| POST | `/v1/admin/spots/ingest` | admin | Enqueue an OSM ingest for a country |
| PATCH | `/v1/admin/spots/:uid` | admin | Publish/reject + curate a spot |
| GET | `/v1/me/favorites` | user | The caller's favorited spots |
| POST | `/v1/me/favorites` | user | Favorite a (published) spot |
| DELETE | `/v1/me/favorites/:spotUid` | user | Unfavorite a spot (any status) |

All routes go through `authenticate` (RFC-0002); `/v1/spots` and `/v1/me/favorites` accept any
authenticated principal (anonymous JWT or Clerk), while `/v1/admin/spots` additionally runs
`requireAdmin`. Responses use the standard `{ data }` success envelope (`HTTPResponse.success`)
and the central `{ error, reason?, message, statusCode }` on failure. Response schemas carry
`.describe()` + `.meta({ ref })`.

**`GET /v1/spots/nearby`** — Auth: user.
- **Request (`query`, `nearbyQuerySchema`):** `lat` (−90..90), `lon` (−180..180), `radiusKm`
  (positive, ≤ 500, default 50), `sport?` (enum), `limit` (int, ≤ 200, default 50). Values are
  `z.coerce`d from the query string.
- **Response:** `NearbySpotResponse` = `{ spots: SpotResponse[] & { distanceKm } }`, nearest
  first; `distanceKm` is the great-circle distance from the query point. Only `published` spots.
- **Errors:** `FORM_ERROR` (422) on invalid params; `UNAUTHENTICATED` (401).
- **Example:** `GET /v1/spots/nearby?lat=38.3&lon=26.4&radiusKm=25&sport=windsurf` →
  `{ "data": { "spots": [ { "uid": "…", "name": "Alaçatı", "distanceKm": 3.2, "shoreBearingDeg": 200, … } ] } }`.

**`GET /v1/spots/search`** — Auth: user.
- **Request (`query`, `searchQuerySchema`):** `q` (2–100 chars), `sport?`, `limit` (≤ 50,
  default 20). The repository escapes LIKE metacharacters (`% _ \`) so a user's literal `%`
  isn't a match-all.
- **Response:** `SpotListResponse` = `{ spots: SpotResponse[] }`, published only.

**`GET /v1/spots/:uid`** — Auth: user.
- **Request (`param`):** `uid` (uuid).
- **Response:** `SpotResponse`. **Errors:** `NOT_FOUND` (404, reason `SPOT_NOT_FOUND`) when the
  spot is missing *or not published* — pending/rejected spots are indistinguishable from
  absent to the public. (RFC-0005 mounts `/v1/spots/:uid/conditions|forecast` alongside.)

**`POST /v1/spots/suggest`** — Auth: user, **rate-limited** (`windowMs: 60_000, max: 10,
keyPrefix: "spot-suggest"`).
- **Request (`json`, `suggestSpotSchema`):** `name` (2–200), `latitude`, `longitude`,
  `country?`/`region?`/`locality?` (≤ 100), `waterType?`, `supportedSports` (≥ 1),
  `goodWindDirections?`/`riskyWindDirections?` (16-point compass arrays — local knowledge is
  exactly what a suggester has; stored on the same curated columns, moderation can refine),
  `notes?` (≤ 500 → `suggestionNotes`, moderator-facing only).
- **Response:** `SpotResponse` with `source=user_suggested`, `status=pending`,
  `createdBy=<caller>`. It is invisible to the public until an admin publishes it. The
  response never echoes `notes` — public `SpotResponse` must not carry moderator-facing text.

**`GET /v1/admin/spots`** — Auth: admin.
- **Request (`query`, `adminSpotQuerySchema`):** `status` (default `pending`), `limit` (≤ 200,
  default 50). **Response:** `AdminSpotListResponse` — `SpotResponse` rows extended with
  `suggestionNotes` (the one surface that shows it). Non-admins get `FORBIDDEN` (403).

**`POST /v1/admin/spots/ingest`** — Auth: admin.
- **Request (`json`, `ingestSpotsSchema`):** `isoCountryCode` — exactly 2 chars, upper-cased by
  a Zod `transform` (so `"tr"` still matches an OSM area). **Response:** `SpotIngestResponse` =
  `{ taskId }`, the enqueued Trigger run id. This does not block on the ingest; it fires the
  background task (§8) and returns immediately.

**`PATCH /v1/admin/spots/:uid`** — Auth: admin.
- **Request (`param` uid + `json`, `moderateSpotSchema`):** every field optional so the same
  endpoint serves publish (`status: "published"`), reject (`status: "rejected"`), and curation
  (`name`, `country`/`region`/`locality` nullable, `waterType`, `skillSuitability`,
  `supportedSports`, `shoreBearingDeg` 0–360, `goodWindDirections`, `riskyWindDirections`,
  `hazards` — the curation-intelligence set). **Response:** the updated `SpotResponse`.
  **Errors:** `NOT_FOUND` (404) when the uid doesn't exist.

**`GET /v1/me/favorites`** — Auth: user. **Response:** `SpotListResponse`, the caller's
favorited spots most-recently-favorited first.

**`POST /v1/me/favorites`** — Auth: user.
- **Request (`json`, `favoriteSpotSchema`):** `spotUid` (uuid). **Response:** the favorited
  `SpotResponse`. **Errors:** `NOT_FOUND` (404, `SPOT_NOT_FOUND`) if the spot is missing or not
  published (you can only favorite a published spot); `ALREADY_EXISTS` (**409** — Splash delta,
  reason `SPOT_ALREADY_FAVORITED`) if already favorited.

**`DELETE /v1/me/favorites/:spotUid`** — Auth: user.
- **Request (`param`):** `spotUid` (uuid). **Response:** `204 No Content`. Unfavorite accepts
  **any** status (a spot later unpublished must still be removable, so the user isn't stranded
  with a dead favorite). **Errors:** `NOT_FOUND` (404, `SPOT_FAVORITE_NOT_FOUND`) if no such
  favorite.

## 7. Services & Business Logic

The layer contract is strict: routes call module services; services (`BaseUseCase`, no DB
handle) orchestrate; repositories (`BaseRepository`, the only holder of `dbClient` and Drizzle
operators) touch the database. A `toSpotResponse(spot)` mapper in `spot.service.ts` is the
single boundary that projects an internal `Spot` row to the public `SpotResponse` — it drops
`id`, `osmId`, `source`, `createdBy`, timestamps (the specs assert `id`/`osmId` never surface).

**`SpotRepository`** (`spot.repository.ts`).
- `findNearby(params)` — the geo hot path. Builds the bbox via `boundingBox(lat, lon,
  radiusKm)`; the SQL distance is
  `EARTH_RADIUS_KM * acos(least(1, greatest(-1, …)))` (the `acos` argument clamped to `[-1, 1]`
  to dodge float-rounding domain errors). The `WHERE` is `status = 'published'` AND latitude in
  `[latMin, latMax]` AND a longitude predicate that OR-s the 1–2 `longitudeRanges(...)` segments
  (antimeridian-safe) AND `distance <= radiusKm` AND, if `sport` is given,
  `arrayContains(supportedSports, [sport])`. Ordered by the distance expression, limited. The
  bbox narrows the index scan; the haversine both orders and enforces the true circular radius —
  no PostGIS (D-003).
- `searchByName(q, limit, sport?)` — `status = 'published'` AND `ILIKE %q%` (with LIKE
  metacharacters escaped) AND optional sport containment.
- `findByUid` / `findByOsmId` / `listByStatus` / `create` / `updateByUid` — CRUD over an
  explicit column allowlist (`spotColumns`, never `SELECT *`, so a future private column can't
  silently surface).
- `bulkInsertOsmPending(values)` — the idempotent seed. Inserts new OSM elements as `pending`;
  `onConflictDoNothing({ target: osmId, where: sql`osm_id IS NOT NULL` })` skips already-seen
  `osmId`s so a re-ingest never clobbers curation edits. The `where` predicate must mirror the
  partial index (§5, §16). Returns the inserted count.

**`SpotService`** (`spot.service.ts`).
- `nearby(query)` / `search(query)` / `detail(uid)` — thin wrappers over the repo + mapper;
  `detail` throws `NOT_FOUND` for a missing *or non-published* spot.
- `suggest(user, input)` — creates `source=user_suggested`, `status=pending`,
  `createdBy=user.id`.
- `listByStatus(status, limit)` / `moderate(uid, input)` — admin reads/updates;
  `moderate` throws `NOT_FOUND` when `updateByUid` returns nothing.
- `requestOsmIngest(isoCountryCode)` — enqueues the Trigger task via `triggerSpotOsmIngest`
  and returns `{ taskId }` (the run id). The service never runs the ingest inline.
- `getGeoByUid(uid)` and `listHotSpotGeos()` — the **seams for weather** (RFC-0005).
  `getGeoByUid` returns the minimal `SpotGeo` (`uid`, `latitude`, `longitude`,
  `shoreBearingDeg`, `supportedSports`) for a *published* spot; `listHotSpotGeos` delegates to
  the favorite repo's distinct-favorited query (the D-004 hot set). `SpotService` satisfies the
  narrow `spotPort` interface weather receives at the composition root — spot's internal ids and
  curation fields never cross the boundary.

**`FavoriteService`** (`favorite.service.ts`).
- `list(user)` — the user's favorited spots as `SpotResponse[]`.
- `add(user, spotUid)` — resolves the spot **requiring published** (a pending/rejected spot
  must not leak into the list or the weather hot set), then `favoriteRepository.add`; a `false`
  return (idempotent conflict) becomes `ALREADY_EXISTS`.
- `remove(user, spotUid)` — resolves the spot allowing **any** status, then
  `favoriteRepository.remove`; a `false` return becomes `NOT_FOUND` (`SPOT_FAVORITE_NOT_FOUND`).

**`SpotIngestService`** (`spot-ingest.service.ts`) + the pure `normalizeOverpassElement`.
- `normalizeOverpassElement(element, isoCountryCode)` — a **pure, independently-unit-tested**
  function (the real ingest IP). Takes coordinates from `element.lat/lon` or `element.center`
  (ways/relations arrive with `out center`), returns `null` if there's no coordinate or no name
  (unnamed POIs aren't presentable). Maps OSM `sport` tags (`windsurfing→windsurf`,
  `kitesurfing→kitesurf`, `sailing→sailing`, `canoe/kayak→kayak`), splits multi-valued
  `sport=a;b` tags, adds `sailing` for marinas / nautical centres / harbours, defaults to
  `["other"]` when nothing maps, sets `waterType=marina` for marinas, and produces
  `osmId = "{type}/{id}"`, `source=osm`, `status=pending`.
- `ingestByCountry(isoCountryCode)` — fetches via the Overpass client, normalizes (dropping
  `null`s), bulk-inserts pending, and returns `{ fetched, normalized, inserted }` (logged). The
  shore-bearing / good-wind enrichment is deliberately **not** done here — it is curation work,
  and the coastline-tangent auto-derivation is P1.

**FavoriteRepository.reassignOwner + the D-008 merge (cross-domain transaction).** This is the
first implementation of the merge-reassign seam. `reassignOwner(fromUserId, toUserId, tx)` runs
on a **caller-supplied executor** typed `DBExecutor` (`PgDatabase<any,any,any>` — satisfied by
both the full client and a transaction handle, defined in `db.manager.ts`). It first **deletes**
the source user's favorites that would collide with the target's existing `(userId, spotId)`
uniques, then **updates** the remaining source favorites to `toUserId`. Wiring: `spot.module.ts`
exposes `favoriteReassigner: MergeReassigner` (`src/types.ts`); the composition root
(`container.ts`) builds spot (and activity) *before* auth, destructures out the reassigners, and
passes `mergeReassigners: [favoriteReassigner, activityReassigner]` explicitly into
`createAuthModule` — auth never imports the feature domains, preserving `platform ↛ feature`. In
`AuthService.linkAnonymousToClerk` branch-2 (target Clerk account already exists), all
reassigners plus `markMergedInto` run inside **one** `userRepository.transaction(tx => …)`, so an
account-link never half-moves owned data. (Branch-1, upgrade-in-place on the same `user.id`,
already preserves everything.) Favorites are the *first* real user data to flow through this seam
([[decisions]] D-008; [[../otonom-kararlar]] §22); activity (RFC-0006) adds a second reassigner
to the same seam.

## 8. Background Jobs (Trigger.dev)

**`spot-osm-ingest`** — the **first Trigger.dev task in the codebase**, and the canonical
pattern every later task copies. Files follow the `<name>.{schema,task,trigger}.ts` convention:

- **`.schema.ts`** — the task id `spot-osm-ingest`, the payload schema (`isoCountryCode`, 2
  chars, upper-cased — normalized here *too*, not only at the route, so a cron/manual `"tr"`
  still matches an OSM area), and the typed `SpotOsmIngestTask`.
- **`.trigger.ts`** — `triggerSpotOsmIngest(isoCountryCode)` calls `tasks.trigger` and returns
  the run id. Invoked from `SpotService.requestOsmIngest` (a **service**, never a route).
- **`.task.ts`** — the `schemaTask`. Config: `maxDuration: 300` (Overpass can take ~90 s for a
  whole country), `retry: { maxAttempts: 3 }`, and — critically — `queue: { concurrencyLimit: 1
  }`, because Overpass (like most external data APIs) rate-limits by IP; **never fan out**. The
  `run` body is the reference lifecycle: `initializeForTrigger()`, then `await
  createDBManagerForTrigger()` (a per-task pool), then **inside `try`** `buildContainer(dbManager)`
  + `services.spotIngestService.ingestByCountry(...)` (wrapped in `logger.trace`); on error it
  reports via `Tracking.captureException`; and **always** `await finalizeTrigger(dbManager)` in
  `finally` so the pool is reset even on failure (no leak). Building the container *inside* `try`
  (not before it) is the review-hardened shape (§16) that guarantees the `finally` runs.

Idempotency/recompute: re-running the same country is safe — `bulkInsertOsmPending` skips
already-seen `osmId`s, so curation survives. A dedupe/update **diff cron** against OSM is a P1
follow-up.

## 9. Dependencies & Integrations

- **OpenStreetMap / Overpass API.** `packages/overpass` `OverpassClient.fetchByCountry` POSTs an
  OverpassQL query (`area["ISO3166-1"=…]` filtered to `sport~windsurfing|kitesurfing|sailing|
  surfing`, `sport=nautical_center`, `leisure~marina|slipway|nautical_center`,
  `seamark:type=harbour`; `out center tags`). The client is a `packages/` HTTP client, **not** a
  repository (it talks to an external API, not our DB), reads the endpoint from
  `globalConfig.config.osm.overpassUrl` lazily at call time, and throws
  `EXTERNAL_SERVICE_ERROR` on transport failure or a non-OK status. **Licensing:** OSM data is
  **ODbL** → attribution ("© OpenStreetMap contributors") is mandatory in the UI. Commercial
  spot guides (Kite&Windsurfing Guide, Global Kite Spots, Windguru/Windy/Windfinder) are
  ToS-protected and **never scraped or copied** — reference-only during curation.
- **Env:** `OSM_OVERPASS_URL` (defaults to `https://overpass-api.de/api/interpreter` in
  `global-config`; declared in `env.d.ts` and `.env.sample`).
- **RFC dependencies:** builds on **RFC-0002** (auth: `c.var.user`, anonymous-vs-Clerk,
  `isAdmin`, `requireAdmin`, `rateLimit`) and RFC-0001 (base classes, container, error/route
  utils, Trigger lifecycle helpers, DB manager).
- **Seams this RFC exposes for later RFCs:** the `SpotGeo` port + `getGeoByUid`/`listHotSpotGeos`
  for **weather (RFC-0005)**; the `favoriteReassigner` merge hook for the **auth merge (D-008)**
  that **activity (RFC-0006)** will extend.

## 10. Security & Privacy

- **Authorization.** Public spot + favorites routes accept any authenticated principal; the
  admin surface adds `requireAdmin`, which **fails closed** — a missing user or `isAdmin=false`
  is `FORBIDDEN` (403, never `UNAUTHORIZED`). Moderation, ingest, and curation are admin-only.
- **Visibility.** Public reads are `published`-only at the repository level; `pending`/`rejected`
  spots (including a user's own just-suggested spot) are `404` to the public, so unmoderated or
  rejected content never leaks. Favoriting requires published; unfavoriting doesn't.
- **User-scoping.** Favorites are keyed to `user.id`; a caller only ever sees/mutates their own.
  The merge reassign (D-008) moves data atomically so an account-link can't orphan or duplicate a
  user's favorites.
- **Input hardening.** Zod bounds every input (lat/lon ranges, `radiusKm ≤ 500`, `limit` caps,
  string lengths, uuid params, 2-char ISO code). `searchByName` escapes LIKE metacharacters so a
  user's `%`/`_`/`\` are literal, not match-all/DoS. Suggest is rate-limited (10/min).
- **Licensing as a compliance control.** The "OSM only, never scrape commercial guides" rule is a
  legal boundary, not a preference — enforced by making Overpass the only ingest source and
  keeping commercial guides reference-only.

## 11. Observability

- **Ingest logging.** `SpotIngestService` logs `OSM ingest complete` with
  `{ isoCountryCode, fetched, normalized, inserted }`; the task wraps the call in
  `logger.trace("ingest-by-country", …)` and logs a `Spot OSM ingest finished` line with the
  result + payload.
- **Error reporting policy** (from RFC-0001). Overpass failures surface as
  `EXTERNAL_SERVICE_ERROR`, which the central error handler *reports* as an exception (it's an
  infra incident worth paging on); the task additionally `Tracking.captureException`s with the
  task id + country before rethrowing (so a retry-exhausting failure is visible). Expected
  `GenericError`s (`NOT_FOUND`, `ALREADY_EXISTS`, `FORM_ERROR`, `FORBIDDEN`) are returned to the
  client without reporting.

## 12. Performance & Scalability

- **Geo hot path.** At the target scale (~30k spots, D-003) the `(latitude, longitude)` btree
  bbox pre-filter + in-SQL haversine ordering is sub-millisecond — no PostGIS, no external geo
  API, no per-request round trips. The bbox keeps the haversine (an `acos` per candidate row)
  off the full table. If the dataset grows into the millions, the documented next step is
  `cube`/`earthdistance` + a GiST index (or PostGIS) — deferred until it's justified.
- **Payloads.** Nearby/search/detail return a compact `SpotResponse` (no timestamps/internal
  ids); `limit` is capped (200 nearby, 50 search) so a single response can't blow up.
- **Ingest.** Bounded to `concurrencyLimit: 1` against Overpass and a per-task DB pool, so
  background seeding never starves the HTTP pool and never trips the external rate limit. A whole
  country is a single bulk insert with `ON CONFLICT DO NOTHING`.
- **Hot set.** The weather pre-warm set is *demand-driven* (favorites), not the whole world
  (D-004), so weather cost scales with interest, not catalogue size.

## 14. Alternatives Considered

- **PostGIS / `cube`+`earthdistance` from day one.** Rejected (D-003). At ~30k rows plain
  Postgres bbox+haversine is sub-ms; PostGIS's install/maintenance cost is unjustified until
  the dataset is millions of rows or needs polygon/complex spatial ops.
- **A third-party geo API for "nearby".** Rejected — nearby is our own DB (map tiles come free
  from MapKit on the client). No external dependency, latency, or cost for the core query.
- **Scraping commercial spot guides** (Kite&Windsurfing Guide ≈ 3500 spots, Global Kite Spots,
  Windguru/Windy/Windfinder). Rejected on licensing — they're ToS-protected and non-scrapable.
  OSM (ODbL, attributed) is the only clean bulk source; the differentiator is the enrichment we
  add on top, not the raw list.
- **Per-spot weather-station references.** Dropped — D-004 makes Open-Meteo the single source
  with no station-confidence claim, so a station field on `spot` would be dead weight.
- **Auto-deriving `shoreBearingDeg` from coastline tangents at ingest.** Deferred to **P1**.
  It needs pulling OSM `natural=coastline` geometry and resolving the tangent's 180°
  land-vs-sea ambiguity into the outward normal; until then curators set the bearing by hand
  ([[../otonom-kararlar]] §23).
- **Storing the on/off/side-shore verdict on the row.** Rejected — it's derived from
  `shoreBearingDeg` + live wind by `windSide()` at query time, so it never goes stale.

## 16. Open Questions & Resolved Decisions

  runtime — invisible to the build because there's no live DB in CI. ✅ (§5, §7)
